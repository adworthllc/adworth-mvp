02 Description of Services

Adworth℠ is a privacy technology platform currently providing:

  • An Ad Intelligence pipeline (adworth-ads Cloudflare Worker v4) that retrieves publicly available advertiser data via SerpApi, analyzes it using Anthropic’s Claude AI, and includes a PII Advertiser Name Classifier that detects and anonymizes individual person names before any response leaves the worker — logging only a non-reversible SHA-256 audit hash with 90-day retention
  • An Ad Intelligence Dashboard at adworth.app/dashboard.html powered by the above pipeline, providing DSA compliance and ad transparency analysis, with inline waitlist signup forms that submit directly to our own Cloudflare Worker via the /submit JSON endpoint
  • A cryptographic consent token API for issuing, validating, and revoking data consent (Patent Component #1)
  • An append-only Invisibility Ledger recording all consent lifecycle and enforcement events (Patent Component #3)
  • A Privacy Governance Engine that enforces consent boundaries in real time by evaluating every data access request (Patent Component #2)
  • A State-Persistence Monitor that scans for unauthorized data retention after consent revocation or expiry (Patent Component #4)
  • A Removal Payload Engine available both within the State-Persistence Monitor and as a standalone worker (adworth-rpe) that generates legally-cited, HMAC-SHA256-signed deletion requests when violations are detected (Patent Component #5) — includes public verification URLs, a demo mode on the enterprise page, and support for five regulatory frameworks (GDPR Article 17, GDPR Article 15, CCPA § 1798.105, CPRA § 1798.100, DSA Article 26)
  • An Ingestion API (adworth-ingestion-api Cloudflare Worker) that receives, cryptographically verifies, and stores Ed25519-signed marketing data submitted by the Privacy Cockpit Chrome extension — private keys are generated on user devices and never transmitted to Adworth
  • A Privacy Cockpit Chrome extension (Manifest V3) that extracts campaign-level metrics from advertising platform dashboards, strips all PII client-side, signs the sanitized data with a device-native Ed25519 keypair, and submits it to the Ingestion API
  • A Consent Widget for users to grant and revoke consent tokens through a visual interface (proof-of-concept demonstration)
  • An Advertiser Integration sandbox for testing data access requests against the Governance Engine (proof-of-concept demonstration)
  • A waitlist for our upcoming cryptographic consent marketplace, processed via our own Cloudflare Worker with an authenticated admin dashboard for waitlist management
  • A CCPA Enforcement Gap page (adworth.app/ccpa-gap) displaying aggregated data from publicly available CCPA transparency reports under California Civil Code § 1798.185(a)(7)
  • Informational content at adworth.app

We are an early-stage company. Our services are actively being developed and may change, be interrupted, or be discontinued at any time. We will make reasonable efforts to notify users of significant changes.

03a Ingestion API & Privacy Cockpit Extension

The following terms govern your use of the Adworth Ingestion API (adworth-ingestion-api) and the Privacy Cockpit Chrome extension.

Cryptographic key architecture

The Privacy Cockpit extension uses Ed25519 asymmetric cryptography (FIPS 186-5 approved) to sign all data submissions. By installing and using the Privacy Cockpit extension, you acknowledge that:

  • An Ed25519 keypair is generated locally on your device using the browser’s native Web Crypto API (crypto.subtle) during first-run setup
  • Your private key is never transmitted to Adworth — it is stored exclusively in chrome.storage.local on your device as a non-extractable key object
  • Only your public key is uploaded to Adworth’s Ingestion API and stored in Cloudflare KV under the key pattern user:{user_id}:pubkey:{device}
  • Adworth uses your public key solely to verify the cryptographic signature on each data submission, confirming the data originated from your registered device
  • Each device you register generates its own keypair — multi-device support is provided by design
  • If your device is lost, compromised, or decommissioned, you may request key revocation by contacting hello@adworth.app — revocation invalidates that device’s ability to submit further data but does not affect previously verified ingestion records
  • Setup tokens issued during the registration flow are single-use and expire after five minutes — they cannot be reused or replayed

Data extraction and sanitization

The Privacy Cockpit extension extracts campaign-level metrics from advertising platform dashboards (including Google Ads and Meta Ads) that you are actively viewing in your browser. By using the extension, you acknowledge that:

  • Data extraction occurs client-side within your browser — Adworth never receives raw dashboard content
  • The extension strips all personally identifiable information (including advertiser billing names, email addresses, and billing addresses) before transmission — only campaign-level metrics are retained (campaign identifiers, spend figures, conversion counts, ROAS, CPC, and similar performance data)
  • Sanitized data is cryptographically signed with your device’s private key before transmission, proving authenticity without exposing private key material
  • You are responsible for ensuring you have appropriate authorization to extract and submit data from the advertising dashboards you access using the extension
  • The extension uses a MutationObserver to detect dashboard updates and may trigger automatic data submissions when dashboard content changes

Ingestion data storage

Verified ingestion records are stored in Cloudflare Workers KV under the key pattern ingestion:{user_id}:{ingestion_id}. Each record contains: sanitized campaign metrics, submission timestamp, device identifier, and a signature_verified: true flag. No personally identifiable information is present in stored ingestion records. All operations against the Ingestion API are audit-logged. Ingestion records are retained for the duration of your account and may be accessed via the authenticated admin dashboard. You may request deletion of your ingestion records by contacting hello@adworth.app.

Current status: The Ingestion API and Privacy Cockpit extension are in active development for investor demonstration and user testing. The architecture described above represents the implemented design. Features, endpoints, and behaviors may change as the platform matures. The extension is currently distributed as a developer build and has not been published to the Chrome Web Store.

04 Consent API & Ledger Terms

The following terms govern your use of Adworth’s consent token system and Invisibility Ledger.

Consent tokens

Consent tokens represent specific, bounded grants of data consent. By issuing a consent token, you acknowledge that:

  • Each token is bound to a specific advertiser, data scope, and time window as specified at the time of issuance
  • Tokens are identified by cryptographically generated random values and are not linked to personally identifiable information
  • Tokens expire automatically at the end of the specified duration and are purged from active storage
  • You may revoke any token at any time; revocation is immediate and irreversible
  • A revoked or expired token cannot be reactivated — a new token must be issued for any subsequent consent grant
  • Revoked and expired tokens may be scanned by the State-Persistence Monitor to detect unauthorized data retention by third parties

Invisibility Ledger

All consent lifecycle events (issuance, validation, revocation, expiration) and enforcement events (access decisions, violation detection, removal payload generation) are recorded in the Invisibility Ledger. By using the consent system, you acknowledge and agree that:

  • Ledger entries are append-only and immutable — once written, they cannot be modified, overwritten, or deleted by any party, including Adworth
  • This immutability is by design and serves as the tamper-evident audit trail for regulatory compliance and dispute resolution
  • Ledger entries contain only cryptographic identifiers, event types, and timestamps — not personally identifiable information
  • Ledger event types include CONSENT_ISSUED, CONSENT_REVOKED, CONSENT_EXPIRED, ACCESS_CHECKED, VIOLATION_DETECTED, and RPE_GENERATED
  • Ledger entries are retained as part of the consent audit trail, even if you request deletion of other personal data
  • The Invisibility Ledger viewer (adworth-ledger) is read-only by architectural design — it cannot create, modify, or delete entries
  • You may request an export of your complete ledger history at any time by contacting hello@adworth.app

Data ownership

Your data remains yours. Adworth’s consent infrastructure does not transfer ownership of your data to Adworth or any third party. The consent token system exists to enforce your decisions about who can access your data, for what purpose, and for how long.

Adworth acts as infrastructure — we facilitate consent enforcement but do not access, use, or monetize the underlying user data that consent tokens govern.

05 State-Persistence Monitor & Removal Payload Engine Terms

The following terms govern your use of Adworth’s State-Persistence Monitor (Patent Component #4) and Removal Payload Engine (Patent Component #5).

State-Persistence Monitor

The State-Persistence Monitor scans for unauthorized data retention by third parties after consent has been revoked or expired. By using or benefiting from this service, you acknowledge that:

  • Scans are initiated via authenticated API request and process only data already present in the Invisibility Ledger
  • The monitor checks a registry of data brokers to determine whether any still retain data associated with revoked or expired consent tokens
  • Violation detection events are recorded in the Invisibility Ledger as VIOLATION_DETECTED entries and are subject to the same immutability and retention rules as all ledger entries
  • Scan results, including violation records and broker names, are stored in a dedicated data store (MONITOR_STATE) for audit and compliance purposes
  • Detection results are probabilistic and represent the monitor’s assessment of unauthorized data persistence, not a guaranteed finding of legal violation
Current status: Broker checks are currently simulated against a representative registry of eight data brokers. The monitor does not connect to live broker APIs and does not transmit any data externally. Simulated results are generated for system validation, investor demonstration, and proof-of-concept purposes only.

Removal Payload Engine

The Removal Payload Engine generates legally-cited deletion requests when violations are detected. It operates both within the State-Persistence Monitor and as a standalone Cloudflare Worker (adworth-rpe). By using or benefiting from this service, you acknowledge that:

  • Removal payloads are generated automatically based on violation detection results and applicable legal citations across five supported regulatory frameworks: GDPR Article 17 (Right to Erasure), GDPR Article 15 (Right of Access), CCPA § 1798.105 (Right to Delete), CPRA § 1798.100 (Right to Know), and DSA Article 26 (Transparency of Online Advertising)
  • Each removal payload is cryptographically signed using HMAC-SHA256 with a server-side secret key, and includes a SHA-256 content hash for tamper detection
  • Notice IDs follow the format RPE-YYYY-XXXXXX and are unique, non-sequential identifiers
  • Payload generation events are recorded in the Invisibility Ledger as RPE_GENERATED entries
  • Removal payloads contain only opaque identifiers, timestamps, legal citations, and broker references — no personally identifiable information
  • The standalone adworth-rpe worker provides public verification endpoints allowing any party to independently verify the integrity and authenticity of a generated notice using its Notice ID, without authentication
  • Adworth does not guarantee that removal payloads constitute legally sufficient deletion requests under any specific jurisdiction; users should consult qualified legal counsel for compliance determinations
  • The legal citations included in removal payloads are informational references and do not constitute legal advice
Current status: Removal payloads are generated with real legal citations, HMAC-SHA256 signatures, and complete data structures, but are not transmitted to actual data brokers. All payloads generated via the State-Persistence Monitor are marked with delivery status SIMULATED. The enterprise page includes an RPE demo that generates clearly-marked demonstration notices prefixed with “DEMO — THIS NOTICE CARRIES NO LEGAL EFFECT / NO COMPANY WILL RECEIVE THIS NOTICE”. No deletion requests have been sent to any third party.

Public verification

The adworth-rpe worker exposes public endpoints that allow anyone with a Notice ID to verify that a removal notice was generated by Adworth and has not been tampered with. Verification checks the HMAC signature and content hash against the stored original. Demo-mode notices are clearly identified in verification responses.

No guarantee of enforcement

Adworth’s State-Persistence Monitor and Removal Payload Engine are designed to assist with post-consent enforcement, but Adworth does not guarantee that all instances of unauthorized data retention will be detected, that data brokers will comply with removal payloads, or that detection or enforcement actions will satisfy specific regulatory requirements. These tools are provided as technical infrastructure to support your data rights. They do not replace legal counsel, regulatory filings, or direct communication with data processors.

06 Privacy Governance Engine Terms

The Privacy Governance Engine (adworth-bouncer) is a real-time consent enforcement service that evaluates data access requests against active consent tokens. By using the Governance Engine, you acknowledge that:

  • Every data access request is subject to six-point validation including token existence, revocation status, expiration, advertiser match, scope match, and cryptographic signature verification
  • Access decisions are final and cannot be overridden manually
  • All access decisions (granted and blocked) are permanently recorded in the Governance Log (GOVERNANCE_LOG) and Invisibility Ledger (ADWORTH_LEDGER)
  • Block reason codes are machine-readable and provided for integration purposes: TOKEN_NOT_FOUND, TOKEN_REVOKED, TOKEN_EXPIRED, ADVERTISER_MISMATCH, SCOPE_MISMATCH, SIGNATURE_INVALID, INVALID_REQUEST
  • The Governance Engine operates in real time and may reject requests due to eventual consistency delays in KV storage propagation
  • Dashboard access requires API key authentication; API keys are stored only in browser sessionStorage and are cleared automatically on tab or browser close — never written to localStorage, cookies, or any server-side system
  • Unauthorized access attempts to the dashboard are logged
