03a Ingestion API & Privacy Cockpit Extension

The following terms govern your use of the Adworth Ingestion API (adworth-ingestion-api) and the Privacy Cockpit Chrome extension.

Cryptographic key architecture

The Privacy Cockpit extension uses Ed25519 asymmetric cryptography (FIPS 186-5 approved) to sign all data submissions. By installing and using the Privacy Cockpit extension, you acknowledge that:

  • An Ed25519 keypair is generated locally on your device using the browser’s native Web Crypto API (crypto.subtle) during first-run setup
  • Your private key is never transmitted to Adworth — it is stored exclusively in chrome.storage.local on your device as a non-extractable key object
  • Only your public key is uploaded to Adworth’s Ingestion API and stored in Cloudflare KV under the key pattern user:{user_id}:pubkey:{device}
  • Adworth uses your public key solely to verify the cryptographic signature on each data submission, confirming the data originated from your registered device
  • Each device you register generates its own keypair — multi-device support is provided by design
  • If your device is lost, compromised, or decommissioned, you may request key revocation by contacting hello@adworth.app — revocation invalidates that device’s ability to submit further data but does not affect previously verified ingestion records
  • Setup tokens issued during the registration flow are single-use and expire after five minutes — they cannot be reused or replayed

Data extraction and sanitization

The Privacy Cockpit extension extracts campaign-level metrics from advertising platform dashboards (including Google Ads and Meta Ads) that you are actively viewing in your browser. By using the extension, you acknowledge that:

  • Data extraction occurs client-side within your browser — Adworth never receives raw dashboard content
  • The extension strips all personally identifiable information (including advertiser billing names, email addresses, and billing addresses) before transmission — only campaign-level metrics are retained
  • Sanitized data is cryptographically signed with your device’s private key before transmission, proving authenticity without exposing private key material
  • You are responsible for ensuring you have appropriate authorization to extract and submit data from the advertising dashboards you access using the extension
  • The extension uses a MutationObserver to detect dashboard updates and may trigger automatic data submissions when dashboard content changes

Ingestion data storage

Verified ingestion records are stored in Cloudflare Workers KV under the key pattern ingestion:{user_id}:{ingestion_id}. Each record contains: sanitized campaign metrics, submission timestamp, device identifier, and a signature_verified: true flag. No personally identifiable information is present in stored ingestion records. You may request deletion of your ingestion records by contacting hello@adworth.app.

Current status: The Ingestion API and Privacy Cockpit extension are in active development for investor demonstration and user testing. The extension is currently distributed as a developer build and has not been published to the Chrome Web Store.

03b Regulation Rules Table

The following terms govern the adworth-rules Cloudflare Worker and the machine-readable regulation rules table it serves to the Removal Payload Engine at runtime.

Purpose and architecture

The Regulation Rules Worker (adworth-rules) is patent-supporting infrastructure that replaces hardcoded legal citations in the RPE with deterministic, runtime-evaluated rules. By using any Adworth service that depends on the RPE, you acknowledge that:

  • Legal citations, compliance deadlines, penalty tiers, controller obligations, and third-party notification requirements included in RPE-generated removal payloads are evaluated at runtime from the versioned rules table served by adworth-rules, not from static strings embedded in the RPE Worker
  • The current rules version is v1.0.0, effective March 5, 2026, covering: GDPR-ART17 (Right to Erasure), GDPR-ART7 (Conditions for Consent), CCPA-1798105 (Right to Delete), CCPA-1798120 (Right to Opt-Out), DSA-ART26 (Transparency of Advertising), and DSA-ART25 (Prohibition of Dark Patterns)
  • Rules are evaluated deterministically — the same input will always produce the same legal citation, deadline, and payload fields for a given rules version. No fuzzy interpretation occurs at runtime

Version control and audit trail

  • The regulation rules table (adworth-rules.json) is version-controlled in the adworthllc/adworth-mvp GitHub repository as the source of truth. Every change is a committed, timestamped, and attributed diff
  • Rules are synced from GitHub to the isolated ADWORTH_RULES Cloudflare KV namespace before taking effect at runtime. The sync constitutes a deliberate human checkpoint between GitHub approval and live enforcement logic
  • The schema_version and effective_date fields in the rules table are recorded in every RPE-generated notice, providing a permanent audit link between any notice and the exact rules version that produced it
  • Each RPE notice stores the rules_version it was generated under. If you later dispute the legal basis of a notice, the rules version embedded in the notice can be retrieved from GitHub commit history to reconstruct the exact ruleset that was applied

Extensibility

The rules table is designed to be extended without requiring RPE code changes. Planned additions include Brazil LGPD, UK GDPR, CAN-SPAM, and U.S. state laws (Virginia VCDPA, Colorado CPA, Connecticut CTDPA). When new regulations are added, these terms will be updated and waitlist members will be notified.

Public endpoints

The adworth-rules Worker exposes public, unauthenticated read endpoints for the rules table: GET /rules (full table), GET /rules/:regulation_id (single regulation), and GET /rules/:regulation_id/:right_id (single right). These endpoints are publicly accessible to support transparency, third-party verification, and regulatory audit. Write and sync endpoints (POST /admin/sync) require API key authentication. No personal data is served by or transmitted to this Worker.

Current status: adworth-rules is live in production at adworth-rules.adworthllc.workers.dev serving rules v1.0.0. The RPE (adworth-rpe v2.2.0) communicates with this Worker via a Cloudflare Service Binding (RULES_WORKER) for secure worker-to-worker communication, with a 5-minute in-memory cache. Admin sync operations currently require manual KV entry; GitHub Actions CI-triggered sync is planned for a future release.

04 Consent API & Ledger Terms

The following terms govern your use of Adworth’s consent token system and Invisibility Ledger.

Consent tokens

Consent tokens represent specific, bounded grants of data consent. By issuing a consent token, you acknowledge that:

  • Each token is bound to a specific advertiser, data scope, and time window as specified at the time of issuance
  • Tokens are identified by cryptographically generated random values and are not linked to personally identifiable information
  • Tokens expire automatically at the end of the specified duration and are purged from active storage
  • You may revoke any token at any time; revocation is immediate and irreversible
  • A revoked or expired token cannot be reactivated — a new token must be issued for any subsequent consent grant
  • Revoked and expired tokens may be scanned by the State-Persistence Monitor to detect unauthorized data retention by third parties

Invisibility Ledger

All consent lifecycle events (issuance, validation, revocation, expiration) and enforcement events (access decisions, violation detection, removal payload generation) are recorded in the Invisibility Ledger. By using the consent system, you acknowledge and agree that:

  • Ledger entries are append-only and immutable — once written, they cannot be modified, overwritten, or deleted by any party, including Adworth
  • This immutability is by design and serves as the tamper-evident audit trail for regulatory compliance and dispute resolution
  • Ledger entries contain only cryptographic identifiers, event types, and timestamps — not personally identifiable information
  • Ledger event types include CONSENT_ISSUED, CONSENT_REVOKED, CONSENT_EXPIRED, ACCESS_CHECKED, VIOLATION_DETECTED, RPE_GENERATED, and MINOR_SHIELD_ACTIVE
  • The Invisibility Ledger intentionally contains synthetic “chaff” entries alongside real consent events (see Section 7a). These entries are indistinguishable to external observers and serve as a privacy protection against pattern-of-life analysis. They do not affect consent enforcement, payout accuracy, or your legal rights
  • Ledger entries are retained as part of the consent audit trail, even if you request deletion of other personal data
  • The Invisibility Ledger viewer (adworth-ledger) is read-only by architectural design — it cannot create, modify, or delete entries
  • You may request an export of your complete ledger history (real entries only; chaff entries are excluded from user-facing exports) at any time by contacting hello@adworth.app

Data ownership

Your data remains yours. Adworth’s consent infrastructure does not transfer ownership of your data to Adworth or any third party. The consent token system exists to enforce your decisions about who can access your data, for what purpose, and for how long.

Adworth acts as infrastructure — we facilitate consent enforcement but do not access, use, or monetize the underlying user data that consent tokens govern.

  • Detection results are probabilistic and represent the monitor’s assessment of unauthorized data persistence, not a guaranteed finding of legal violation

06 Privacy Governance Engine Terms

The Privacy Governance Engine (adworth-bouncer) is a real-time consent enforcement service that evaluates data access requests against active consent tokens. By using the Governance Engine, you acknowledge that:

  • Every data access request is subject to six-point validation including token existence, revocation status, expiration, advertiser match, scope match, and cryptographic signature verification
  • Access decisions are final and cannot be overridden manually

07a Advanced Privacy Protocols

Adworth implements the following systemic privacy protocols, active by default across the infrastructure. These mechanisms protect users from external surveillance and are not used to obscure user rights, limit data access, or affect payout accuracy. Full technical disclosure is available in the Privacy Policy Section 8.

Ledger Chaff — noise and decoy injection

The adworth-ledger Worker injects synthetic “chaff” (decoy) entries alongside real consent events. By using the consent system and Invisibility Ledger, you acknowledge that:

  • The ledger intentionally contains synthetic noise entries that are mathematically indistinguishable from real entries to external observers. Only your Sovereign Key can distinguish real from chaff
  • Chaff entries do not affect consent enforcement decisions — the Governance Engine evaluates only cryptographically verified real tokens
  • Chaff entries do not affect payout accuracy — marketplace payments are calculated from verified real tokens only
  • Chaff entries do not affect your legal rights or the enforceability of your real consent records
  • User-facing ledger exports contain real entries only; chaff entries are excluded from exports provided to users upon request
  • This is a deliberate privacy protection consistent with established differential privacy practices for audit log protection, not a data inaccuracy

Local Differential Privacy — 2% data fuzz

A 2% randomized fuzz coefficient is applied to numeric metadata fields in consent scope records where mathematically applicable. By using the consent system, you acknowledge that:

  • Numeric metadata (e.g. time window durations, frequency parameters) in consent scope records may be subject to a 2% fuzz applied within the Worker before data is written
  • This fuzz is never applied to payment amounts, token cryptographic signatures, or legal notice content
  • Individual consent grants remain fully enforceable and auditable; the fuzz affects only aggregate inference risk, not individual record accuracy
  • This technique is a standard implementation of local differential privacy and is disclosed here as a transparency obligation

Location Grid Quantization — spatial privacy

When consent token scope metadata includes a location parameter, coordinates are snapped to a 1 km² grid square. By using location-scoped consent tokens, you acknowledge that:

  • Raw GPS coordinates are never stored or transmitted by Adworth systems
  • Location data in consent records is stored at 1 km² grid precision only, sufficient for geographic scoping but insufficient to identify a specific address
  • This prevents geospatial fingerprinting by third parties attempting to reconstruct location patterns from consent records

Minor Shield Protocol

When a consent token carries user_type: minor metadata, the system enters a High-Shield state. By using services that involve minor users, you acknowledge that:

  • The Governance Engine will block all advertising-related data access requests, returning MINOR_SHIELD_ACTIVE, regardless of token validity or scope
  • This block is final and cannot be overridden by any API call, token parameter, or operator instruction
  • No long-term behavioral profile is built or retained for minor users
  • Synthetic chaff entry volume is increased for minor-associated token IDs as an additional protection
  • Consent tokens associated with minor users cannot be used as the basis for any marketplace payment
  • Adworth does not knowingly collect personal data from users under 13. Minor Shield applies as an additional protection layer for users aged 13–17 where parental or guardian consent exists

Nashville Offline Protocol — roadmap only

A future capability for device-bound offline consent authorization via Secure Enclave and NFC/Bluetooth Mesh Promissory Tokens. This feature is not yet deployed.
