The short version: Adworth is built on the principle that your data belongs to you. We collect only what we need to operate the service, we never sell your data, and you can delete it at any time. Our Ad Intelligence Dashboard retrieves public ad data via SerpApi and analyzes it using Anthropic’s AI — no personal user data is involved in that pipeline. Our consent token API uses opaque identifiers only — no personally identifiable information is stored in tokens or ledger entries. Our Privacy Governance Engine automatically enforces consent boundaries by evaluating every data access request against active tokens in real time, blocking unauthorized access before it occurs. Our State-Persistence Monitor scans for unauthorized data retention after you revoke consent, and our Removal Payload Engine generates legally-cited deletion notices on your behalf. Both currently operate in simulated mode for demonstration purposes. All API access is authenticated and encrypted. API keys entered in any Adworth dashboard are stored in sessionStorage only and are cleared automatically when you close the browser tab — they are never written to permanent storage, cookies, or transmitted beyond your authenticated session. We do not use third-party trackers, advertising pixels, or analytics services. That’s kind of the whole point.

05 Consent Infrastructure & Ledger

Adworth’s patent-pending consent infrastructure processes data through two primary systems: the Consent Token API and the Invisibility Ledger.

Consent Token API

The Consent Token API (adworth-consent-api.adworthllc.workers.dev) manages the complete lifecycle of cryptographic consent tokens. Each token represents a specific, bounded grant of consent from a user to an advertiser.

  • Token issuance: When consent is granted, a 128-bit cryptographically random token identifier is generated using the Web Crypto API. The token records the advertiser, data scope, issuance timestamp, expiration date, and an HMAC-SHA256 cryptographic signature binding all parameters together.
  • Token validation: Any authenticated party can verify whether a consent token is currently valid, expired, or revoked. Validation performs six checks in sequence: token existence, revocation status, expiration, advertiser match, scope match, and cryptographic signature verification.
  • Token revocation: Users can revoke consent at any time. Revocation is immediate and irreversible — a revoked token cannot be reactivated. The revocation timestamp is recorded in both ADWORTH_TOKENS and ADWORTH_LEDGER.
  • Token expiration: Tokens automatically expire at the end of the specified consent duration. Cloudflare KV’s TTL mechanism purges expired tokens from active storage. A 24-hour buffer is added to the TTL to preserve audit integrity during the expiration window.

API authentication

All Consent API endpoints (except the public /health check) require authentication via an X-API-Key header or Bearer token in the HTTP Authorization header. API secret keys are:

  • Stored as encrypted environment secrets within Cloudflare’s Worker infrastructure — never in source code or version control
  • Never exposed in API responses, logs, or client-side code
  • Validated server-side on every request before any data is accessed or returned
  • Rotatable at any time without affecting stored consent data or ledger entries
  • When entered in the Consent API dashboard, stored only in browser sessionStorage for the duration of the session — cleared automatically on tab or browser close, never written to localStorage, cookies, or any persistent storage

Unauthenticated requests to protected endpoints receive a 401 Unauthorized response with no data disclosure.

Invisibility Ledger

The Invisibility Ledger is an append-only, tamper-evident log of all consent lifecycle events stored in the ADWORTH_LEDGER KV namespace. By design, ledger entries cannot be modified or deleted after creation — this ensures a permanent audit trail for regulatory compliance and dispute resolution.

  • What is recorded: Each ledger entry contains an entry ID, event type (e.g. CONSENT_ISSUED, CONSENT_REVOKED, ACCESS_CHECKED, VIOLATION_DETECTED, RPE_GENERATED), the associated token ID, advertiser ID, consent scope, access decision result and reason code (where applicable), and a UTC timestamp.
  • What is not recorded: No email addresses, names, IP addresses, device identifiers, or other personally identifiable information is stored in the ledger.
  • Immutability: Ledger entries are written once and never overwritten. The ledger retains the most recent 500 events in rolling storage. For compliance deployments, indefinite retention can be configured.
  • Access: Ledger data is accessible only through authenticated API requests or the authenticated Ledger viewer dashboard.

Data isolation

Consent token data, ledger entries, monitor state, and governance decisions are stored in dedicated, isolated Cloudflare Workers KV namespaces (ADWORTH_TOKENS, ADWORTH_LEDGER, MONITOR_STATE, and GOVERNANCE_LOG) that are separate from all other Adworth data stores (waitlist, ad cache). There is no cross-contamination between these systems.

06 Privacy Governance Engine

Adworth’s Privacy Governance Engine (Patent Component #2, adworth-bouncer.adworthllc.workers.dev) is a real-time consent enforcement system that automatically evaluates every data access request before any user data can be accessed.

How it works

When an advertiser or third party attempts to access user data, they must first submit an evaluation request to the Governance Engine. The engine performs six validation checks in sequence:

  1. Token existence: Does a consent token exist for this token ID?
  2. Revocation check: Has the user revoked their consent?
  3. Expiration check: Has the consent window expired?
  4. Advertiser match: Does the requesting advertiser match the token holder? (Consent is non-transferable.)
  5. Scope match: Does the requested data type match the scope of the consent token?
  6. Cryptographic signature verification: Is the token’s HMAC-SHA256 signature valid, confirming the token has not been tampered with?

If any check fails, the request is immediately blocked with a specific reason code. If all checks pass, the request is granted. The decision is final and cannot be overridden.

Access decision logging

Every access decision is recorded in two places:

  • The Governance Log (GOVERNANCE_LOG KV namespace): Contains the full decision record including decision identifier, timestamp, advertiser ID, user ID, data scope, decision result, block reason (if applicable), and all six check results.
  • The Invisibility Ledger (ADWORTH_LEDGER KV namespace): Contains an ACCESS_CHECKED event entry recording the decision for the tamper-evident audit trail.

No personally identifiable information is stored in access decision records. All identifiers are opaque.

Block reason codes

When access is blocked, the response includes one of these machine-readable reason codes:

  • TOKEN_NOT_FOUND — No consent token exists for this token ID
  • TOKEN_REVOKED — User has explicitly revoked consent
  • TOKEN_EXPIRED — Consent window has closed
  • ADVERTISER_MISMATCH — Token was issued to a different advertiser
  • SCOPE_MISMATCH — Requested data type is not covered by the consent token
  • SIGNATURE_INVALID — Cryptographic signature verification failed; token may have been tampered with
  • INVALID_REQUEST — Required fields missing from the request

Governance Engine and Consent API dashboards

The Privacy Governance Engine and Consent API each include interactive dashboards accessible at their respective worker /dashboard endpoints. Both dashboards require API key authentication. The following data handling practices apply to all Adworth internal dashboards:

  • API keys are stored in sessionStorage only — cleared automatically when the browser tab or window is closed
  • API keys are never written to localStorage, cookies, or any server-side log
  • All displayed data is fetched in real time from authenticated API endpoints — nothing is cached in the browser
  • Dashboards load no third-party scripts, set no cookies, and use no analytics or tracking technologies
  • Dashboard interfaces are served directly from Cloudflare Workers with no external hosting dependencies

Authentication

All Governance Engine endpoints (except the public health check) require X-API-Key header or Bearer token authentication. The same security practices described in Section 05 apply.

07 State-Persistence Monitor & Removal Payload Engine

Adworth’s patent-pending infrastructure includes two enforcement components that operate after consent is revoked or expires.

State-Persistence Monitor (Patent Component #4)

The State-Persistence Monitor (adworth-state-monitor.adworthllc.workers.dev) is designed to detect unauthorized data retention by third parties after a user revokes or expires their consent. It operates as follows:

  • Scan trigger: Scans are initiated via authenticated API request. Each scan reads the Invisibility Ledger to identify all revoked and expired consent tokens.
  • Broker checks: For each revoked or expired token, the monitor checks a registry of data brokers to determine whether any still retain the user’s data.
  • Violation detection: If a broker is found to be retaining data beyond the consent window, a VIOLATION_DETECTED event is recorded in the Invisibility Ledger.
  • Data processed: Only opaque token identifiers, user IDs, advertiser IDs, consent scopes, and timestamps are processed. No personally identifiable information is involved.
Current status: Broker checks are currently simulated against a registry of eight representative data brokers. The monitor does not yet connect to live broker APIs or transmit any data externally. Simulated results are used for system validation, investor demonstration, and architecture proof-of-concept purposes only.

Removal Payload Engine (Patent Component #5)

When the State-Persistence Monitor detects a violation, the Removal Payload Engine automatically generates a legally-cited deletion request. Each removal payload includes:

  • Recipient information: The name and type of the data broker found retaining data
  • Data subject reference: Opaque user ID and token ID only — no PII
  • Consent record: Issuance date, revocation date, elapsed time since revocation, original consent scope, and advertiser ID
  • Legal citations: Applicable regulations including GDPR Article 17 (Right to Erasure), CCPA § 1798.105 (Right to Delete), and DSA Article 26 (Advertising Transparency), with specific compliance deadlines
  • Cryptographic proof: Ledger reference and token signature linking back to the tamper-evident audit trail in ADWORTH_LEDGER
  • Required action: Deletion deadline and confirmation endpoint

Each removal payload generation event is recorded in the Invisibility Ledger as an RPE_GENERATED entry, creating a complete audit trail of enforcement actions.

Current status: Removal payloads are generated with real legal citations and complete data structures, but are not yet transmitted to actual data brokers. Delivery status is marked as SIMULATED in all current payloads. No deletion requests have been sent to any third party. When live delivery begins, this policy will be updated to reflect the transmission methods, recipient brokers, and any additional data sharing involved.

Monitor data storage

Scan results, violation records, and removal payload metadata are stored in a dedicated Cloudflare Workers KV namespace (MONITOR_STATE). This namespace is separate from all other Adworth data stores. Violation detection events and RPE generation events are additionally recorded in the ADWORTH_LEDGER namespace as part of the permanent, tamper-evident audit trail.

Monitor dashboard

The State-Persistence Monitor includes an interactive dashboard accessible at the monitor worker’s /dashboard endpoint. This dashboard requires API key authentication. The same data handling practices described in Section 06 apply: no cookies, no localStorage, no third-party scripts, and API keys held in sessionStorage only.

08 Data Retention

We retain your personal information only as long as necessary to provide our services or as required by law.

  • Waitlist email addresses are stored in Cloudflare Workers KV (WAITLIST namespace) until you unsubscribe or request deletion.
  • Ad intelligence cache (SerpApi + Anthropic results) is stored temporarily in Cloudflare Workers KV (AD_CACHE namespace) with a TTL of 1–2 hours and then automatically purged.
  • Consent tokens are stored in Cloudflare Workers KV (ADWORTH_TOKENS namespace) with a TTL matching the consent duration plus a 24-hour audit buffer. Tokens are automatically purged by Cloudflare after expiration. Revoked tokens retain their REVOKED status until their original expiration date, then are purged.
  • Consent ledger entries are retained on a rolling basis of the most recent 500 events in Cloudflare Workers KV (ADWORTH_LEDGER namespace). These entries contain only cryptographic identifiers and timestamps — not personal information. Extended or indefinite retention can be configured for regulatory compliance deployments.
  • Monitor scan results are stored in Cloudflare Workers KV (MONITOR_STATE namespace) and retained for audit and compliance purposes. These records contain only opaque identifiers, timestamps, and broker names — no PII.
  • Governance Engine decisions are stored in Cloudflare Workers KV (GOVERNANCE_LOG namespace) and retained for audit and compliance purposes. Decision records include access evaluation results, advertiser and user identifiers, data scope, and block reason codes. No PII.
  • Violation and RPE ledger entries (VIOLATION_DETECTED and RPE_GENERATED events) are retained in the ADWORTH_LEDGER namespace following the same rolling retention policy as consent lifecycle entries.

When you request deletion of your personal data, we will remove it from our active systems within 30 days and from backup systems within 90 days. Note that consent ledger entries, violation records, and RPE records (which contain no PII) may be retained as part of the immutable audit trail even after personal data deletion, as they serve as proof of consent actions and enforcement for both users and advertisers.

10 Security

We implement industry-standard security measures to protect your information, including:

  • Encrypted data transmission (HTTPS enforced via Cloudflare on all endpoints and worker subdomains)
  • Cryptographic generation of consent token identifiers using the Web Crypto API (crypto.getRandomValues producing 128-bit random values)
  • HMAC-SHA256 cryptographic signing of every consent token, binding the token ID, advertiser, scope, and time window into a tamper-evident signature
  • Bearer token / X-API-Key authentication on all protected API endpoints across all workers
  • API secret keys stored as encrypted environment secrets within Cloudflare’s Worker infrastructure — never in source code, version control, or client-side code
  • API keys entered in Adworth dashboards stored only in browser sessionStorage — never written to localStorage, cookies, or any server-side system
  • Append-only ledger design preventing tampering with consent history, violation records, and enforcement actions
  • Data isolation between all KV namespaces — tokens, ledger, monitor state, governance log, waitlist, and cache are fully separated with no cross-access
  • Six-point cryptographic validation on every data access request ensuring consent cannot be faked, extended, transferred, or misused
  • Real-time consent enforcement via the Privacy Governance Engine, blocking unauthorized access before data can be reached
  • Cryptographic proof references in all removal payloads linking back to the tamper-evident ledger

No method of transmission over the internet is 100% secure. If you believe your information has been compromised, please contact us immediately at hello@adworth.app.
