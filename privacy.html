The short version: Adworth is built on the principle that your data belongs to you. We collect only what we need to operate the service, we never sell your data, and you can delete it at any time. Our Ad Intelligence pipeline runs entirely through our own adworth-ads Cloudflare Worker, which retrieves public ad data via SerpApi and analyzes it using Anthropic’s AI. As of March 2026, adworth-ads includes a PII Advertiser Name Classifier that detects individual person names in returned ad data, anonymizes them before any response leaves the worker, and logs only a non-reversible SHA-256 audit hash — no personal data ever leaves the pipeline. Our consent token API uses opaque identifiers only — no personally identifiable information is stored in tokens or ledger entries. Our Privacy Governance Engine automatically enforces consent boundaries by evaluating every data access request against active tokens in real time, blocking unauthorized access before it occurs. Our State-Persistence Monitor scans for unauthorized data retention after you revoke consent, and our Removal Payload Engine generates legally-cited deletion notices on your behalf. Both currently operate in simulated mode for demonstration purposes. All API access is authenticated and encrypted. API keys entered in any Adworth dashboard are stored in sessionStorage only and are cleared automatically when you close the browser tab. Waitlist submissions are processed entirely by our own Cloudflare Worker at adworth.app/waitlist-submit — no third-party form processors are used. Vercel Analytics is disabled on our deployment. We do not use third-party trackers, advertising pixels, or analytics services. That’s kind of the whole point.

Information processed through the Ad Intelligence pipeline

When you use our Ad Intelligence Dashboard (adworth.app/dashboard.html), all processing runs through our own adworth-ads Cloudflare Worker. The following data is processed:

  • The domain name you enter for analysis (e.g. apple.com)
  • Publicly available ad data retrieved from Google’s Ads Transparency Center via SerpApi — this includes advertiser names, ad creative IDs, formats, campaign dates, and targeting domains
  • AI-generated analysis of that public ad data, produced by Anthropic’s Claude API

No personal user data is transmitted to SerpApi or Anthropic as part of dashboard analysis. The pipeline processes publicly available advertiser data only. Results may be cached temporarily in Cloudflare Workers KV (AD_CACHE namespace) for up to 2 hours.

PII Advertiser Name Classifier (adworth-ads v4 — March 2026)

As of March 2026, the adworth-ads worker includes a PII Advertiser Name Classifier. This component:

  • Scans all advertiser names returned from SerpApi before any response leaves the worker
  • Detects individual person names embedded in ad data (e.g. a sole trader advertising under their own name)
  • Anonymizes detected person names — replacing them with a generic placeholder — before the response is returned to the dashboard or any downstream system
  • Logs a non-reversible SHA-256 audit hash of each detected name to Cloudflare Workers KV with a 90-day retention period — the hash cannot be used to reconstruct the original name
  • Reports the count of anonymized names in each response’s metadata field for transparency

The classifier ensures that individual person names in public ad data are never surfaced to end users or stored in recoverable form anywhere in the Adworth pipeline. Only the audit hash count is retained for compliance tracking.

Information collected through our Consent API

When you interact with our Consent Token API (adworth-consent-api.adworthllc.workers.dev), we collect and process:

  • Consent token identifiers (cryptographically generated 128-bit random values — not linked to your real-world identity)
  • Opaque user identifiers (user IDs you provide; not validated against any identity system)
  • Advertiser identifiers associated with each consent grant
  • Consent scope parameters (data type, time window) as specified by you
  • Timestamps of consent issuance, validation, revocation, and expiry (UTC ISO-8601)
  • HMAC-SHA256 cryptographic signatures binding each token to its parameters — stored as hex strings, not reversible to the underlying secret key
  • Ledger entries recording each consent lifecycle event in the append-only, tamper-evident Invisibility Ledger

Consent tokens contain no personally identifiable information. Token identifiers are 128-bit cryptographic random values only. User IDs are opaque strings with no connection to email addresses, names, or other PII.

Information processed through the Consent API Dashboard

The Consent API dashboard (adworth-consent-api.adworthllc.workers.dev/dashboard) requires API key authentication. API keys are stored in sessionStorage only, cleared automatically on tab or browser close, and never written to localStorage, cookies, or any server-side log. The dashboard loads no third-party scripts and sets no cookies.

Information processed through the Invisibility Ledger dashboard

The Invisibility Ledger dashboard (adworth-ledger.adworthllc.workers.dev/dashboard) is a read-only viewer of the unified tamper-evident audit trail. It reads from ADWORTH_LEDGER KV and never writes to it. It displays the complete event timeline across all five patent components — consent issuance, governance decisions, violations, and removal payloads — with filtering by event type and user. The same sessionStorage-only API key handling applies as all other Adworth dashboards.

Information processed through the State-Persistence Monitor

When the State-Persistence Monitor (adworth-state-monitor.adworthllc.workers.dev) runs a scan, it processes revoked and expired consent token identifiers from the Invisibility Ledger, associated opaque user IDs, advertiser IDs, consent scope data, and scan metadata including violation detection results and removal payload records.

Current status: The State-Persistence Monitor currently uses simulated broker checks for demonstration and investor review purposes. It does not yet connect to live data broker APIs. Removal payloads are generated with real legal citations but are not yet transmitted to actual data brokers.

Information processed through the Privacy Governance Engine

When the Privacy Governance Engine (adworth-bouncer.adworthllc.workers.dev) evaluates a data access request, it processes the requesting advertiser’s identifier, the opaque user identifier, the data scope being requested, and the active consent tokens for that user-advertiser pair. Every decision — granted or blocked — is recorded in both GOVERNANCE_LOG and ADWORTH_LEDGER. No PII is processed or stored by the Governance Engine.

Information processed through the Consent Widget and Advertiser Integration

Adworth provides two demonstration interfaces for investor review and system validation. Neither is publicly deployed or linked from the main Adworth website:

  • Consent Widget: User-facing interface for granting and revoking consent tokens. Simulated payment amounts are displayed for demonstration only — no real financial transactions occur.
  • Advertiser Integration Sandbox: Developer-facing interface for testing access requests against the Privacy Governance Engine. No real advertiser data or production systems are involved.

Information we do not collect

We do not collect government IDs, financial account numbers, or health data. We do not use Formspree or any third-party form processing service — all form submissions are handled exclusively by our own adworth-waitlist Cloudflare Worker. API keys used to access Adworth dashboards are held in sessionStorage only and are never retained on our servers.

No personal user data is transmitted to SerpApi. Before any SerpApi response is returned to the dashboard, the PII Advertiser Name Classifier runs to detect and anonymize individual person names. Results are cached in AD_CACHE KV for up to 2 hours. SerpApi’s privacy policy: serpapi.com/privacy.

Anthropic API — AI privacy analysis

Privacy risk scores, transparency scores, regulatory flags, and AI findings are generated by Anthropic’s Claude AI via the Anthropic API. Data transmitted to Anthropic is the already-anonymized ad metadata from SerpApi (advertiser names, ad counts, formats, dates) — any person names have been replaced by the PII classifier before this call is made. No personal user data, waitlist emails, or consent token data is transmitted to Anthropic. AI analysis results are generated on demand and may be temporarily cached. Anthropic’s privacy policy: anthropic.com/legal/privacy.

05 Consent Infrastructure & Ledger

Consent Token API

The Consent Token API (adworth-consent-api.adworthllc.workers.dev) manages the complete lifecycle of cryptographic consent tokens. On issuance, a 128-bit cryptographically random token is generated via the Web Crypto API and bound to a specific advertiser, data scope, and time window via HMAC-SHA256 signature. Validation performs six sequential checks: token existence, revocation status, expiration, advertiser match, scope match, and cryptographic signature verification. Revocation is immediate and irreversible. Expired tokens are auto-purged by Cloudflare KV TTL with a 24-hour audit buffer.

API authentication

All protected endpoints require an X-API-Key header or Bearer token. Keys are stored as encrypted Cloudflare environment secrets, never in source code or version control. When entered in dashboards, keys are held in sessionStorage only — cleared on tab close, never written to localStorage or cookies. Unauthenticated requests receive 401 Unauthorized with no data disclosure.

Invisibility Ledger

The Invisibility Ledger (ADWORTH_LEDGER KV namespace) is an append-only, tamper-evident log of all consent lifecycle events across all five patent components. Each entry records: entry ID, event type (CONSENT_ISSUED, CONSENT_REVOKED, ACCESS_CHECKED, VIOLATION_DETECTED, RPE_GENERATED), token ID, advertiser ID, consent scope, decision result and reason code where applicable, and UTC timestamp. No email addresses, names, IP addresses, or PII of any kind is stored. Entries are written once and never overwritten. The ledger retains the most recent 500 events in rolling storage; indefinite retention is configurable for compliance deployments.

The adworth-ledger worker provides a read-only dashboard viewer of the complete event timeline with filtering by event type and user identifier. This worker reads from ADWORTH_LEDGER only and has no write access by design, ensuring the viewer cannot tamper with the audit trail.

Data isolation

All KV namespaces (ADWORTH_TOKENS, ADWORTH_LEDGER, MONITOR_STATE, GOVERNANCE_LOG, WAITLIST, AD_CACHE) are fully isolated. There is no cross-namespace access between systems.

06 Privacy Governance Engine

The Privacy Governance Engine (adworth-bouncer.adworthllc.workers.dev) is a real-time consent enforcement system. Every data access request is evaluated against active consent tokens before any data can be reached.

Six-point validation

  1. Token existence — Does a consent token exist for this token ID?
  2. Revocation check — Has the user revoked consent?
  3. Expiration check — Has the consent window closed?
  4. Advertiser match — Does the requester match the token holder? Consent is non-transferable.
  5. Scope match — Does the requested data type match the consent scope?
  6. Cryptographic signature verification — Is the HMAC-SHA256 signature valid?

Any failed check triggers an immediate block. The decision is final and cannot be overridden. Every decision is recorded in both GOVERNANCE_LOG and ADWORTH_LEDGER. No PII is stored in any decision record.

Block reason codes

Blocked responses include one of: TOKEN_NOT_FOUND, TOKEN_REVOKED, TOKEN_EXPIRED, ADVERTISER_MISMATCH, SCOPE_MISMATCH, SIGNATURE_INVALID, or INVALID_REQUEST.

Dashboard

The Governance Engine dashboard (adworth-bouncer.adworthllc.workers.dev/dashboard) displays the real-time decision log, block reason breakdown, and validation stats. API key stored in sessionStorage only. No third-party scripts. No cookies.

07 State-Persistence Monitor & Removal Payload Engine

State-Persistence Monitor (Patent Component #4)

The State-Persistence Monitor (adworth-state-monitor.adworthllc.workers.dev) detects unauthorized data retention after a user revokes or expires consent. On scan, it reads the Invisibility Ledger to identify all revoked/expired tokens, checks a registry of data brokers for unauthorized retention, and records any VIOLATION_DETECTED events in the Invisibility Ledger. Only opaque identifiers, scopes, and timestamps are processed — no PII.

Current status: Broker checks are currently simulated against eight representative brokers. The monitor does not yet connect to live broker APIs or transmit any data externally. Simulated results are used for system validation and investor demonstration only.

Removal Payload Engine (Patent Component #5)

When a violation is detected, the RPE automatically generates a legally-cited deletion request containing: the broker name, opaque user ID and token ID (no PII), consent issuance and revocation dates and elapsed time, original scope and advertiser ID, legal citations (GDPR Article 17, CCPA § 1798.105, DSA Article 26) with compliance deadlines, and a cryptographic proof reference linking to ADWORTH_LEDGER. Each RPE_GENERATED event is recorded in the Invisibility Ledger.

Current status: Removal payloads are generated with real legal citations but are not yet transmitted to actual data brokers. Delivery status is marked SIMULATED. This policy will be updated before live delivery begins.

Monitor data storage

Scan results, violation records, and RPE metadata are stored in the isolated MONITOR_STATE KV namespace. Violation and RPE events are additionally recorded in ADWORTH_LEDGER as part of the permanent audit trail. The monitor dashboard (adworth-state-monitor.adworthllc.workers.dev/dashboard) uses sessionStorage-only API key handling and no third-party scripts.

08 Data Retention

  • Waitlist email addresses — stored in WAITLIST KV until you unsubscribe or request deletion
  • Ad intelligence cache — stored in AD_CACHE KV with a 1–2 hour TTL, then automatically purged
  • PII classifier audit hashes — non-reversible SHA-256 hashes of detected advertiser person names, stored in AD_CACHE KV with a 90-day retention period, then automatically purged. These hashes cannot be used to reconstruct the original name.
  • Consent tokens — stored in ADWORTH_TOKENS KV with a TTL matching the consent duration plus a 24-hour audit buffer, then automatically purged. Revoked tokens retain REVOKED status until their original expiration date.
  • Consent ledger entries — retained on a rolling basis of the most recent 500 events in ADWORTH_LEDGER KV. Entries contain only cryptographic identifiers and timestamps — no PII. Extended or indefinite retention is configurable for compliance deployments.
  • Monitor scan results — stored in MONITOR_STATE KV for audit and compliance purposes. Records contain only opaque identifiers, timestamps, and broker names — no PII.
  • Governance Engine decisions — stored in GOVERNANCE_LOG KV for audit and compliance purposes. Records include access evaluation results, advertiser and user identifiers, data scope, and block reason codes — no PII.
  • Violation and RPE ledger entries — retained in ADWORTH_LEDGER following the same rolling 500-event policy as consent lifecycle entries.

When you request deletion of your personal data, we will remove it from active systems within 30 days and from backups within 90 days. Ledger entries, violation records, and RPE records (which contain no PII) may be retained as part of the immutable audit trail even after personal data deletion, as they serve as proof of consent actions for both users and advertisers.

10 Security

We implement industry-standard security measures including:

  • HTTPS enforced via Cloudflare on all endpoints and worker subdomains
  • 128-bit cryptographic random token generation via the Web Crypto API
  • HMAC-SHA256 cryptographic signing of every consent token
  • Bearer token / X-API-Key authentication on all protected API endpoints
  • API secret keys stored as encrypted Cloudflare environment secrets — never in source code or version control
  • API keys entered in dashboards stored only in browser sessionStorage — cleared on tab close, never written to localStorage or cookies
  • PII Advertiser Name Classifier anonymizing person names before any data leaves the adworth-ads worker; only a non-reversible SHA-256 audit hash is retained
  • Append-only ledger design preventing tampering with consent history, violation records, and enforcement actions
  • Complete KV namespace isolation across all data stores
  • Six-point cryptographic validation on every data access request
  • Real-time consent enforcement via the Privacy Governance Engine
  • Cryptographic proof references in all removal payloads linking to the tamper-evident ledger
  • Waitlist form protected by origin validation, honeypot bot detection, duplicate checking, and email format validation

No method of transmission over the internet is 100% secure. If you believe your information has been compromised, contact us immediately at hello@adworth.app.
